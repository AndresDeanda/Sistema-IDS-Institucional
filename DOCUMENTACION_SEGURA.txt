1. ARQUITECTURA GENERAL DEL SISTEMA

El Sistema IDS Institucional es una herramienta desarrollada en Python para el monitoreo de tráfico de red dentro de una infraestructura controlada. Su objetivo principal es detectar eventos relacionados con seguridad, como dispositivos no autorizados, conexiones hacia direcciones IP peligrosas, consultas a dominios y generación de alertas para el administrador.

El sistema funciona mediante una arquitectura modular. Cada módulo realiza una tarea específica, lo que permite que el programa sea más fácil de mantener, revisar y modificar. El archivo principal ids_main.py se encarga de inicializar todos los módulos y coordinar la ejecución general del IDS.

Componentes principales:

* ids_main.py: archivo principal que inicia el sistema.
* config_loader.py: carga las variables de configuración desde el archivo .env.
* whitelist.py: valida direcciones IP y MAC autorizadas.
* packet_capture.py: captura paquetes de red mediante Scapy.
* traffic_monitor.py: registra actividad de tráfico, dominios y consultas DNS.
* threat_intel.py: compara direcciones IP con una lista negra.
* email_alerter.py: envía alertas por correo electrónico.
* forensics.py: realiza consultas de apoyo forense sobre IPs sospechosas.
* logger.py: genera registros del sistema y bitácoras de ejecución.
* config/whitelist.csv: almacena dispositivos autorizados.
* config/blacklist.txt: almacena direcciones IP consideradas peligrosas.
* env.example: plantilla de variables de configuración sin credenciales reales.

El sistema está diseñado para ejecutarse en un entorno autorizado, como una red institucional, laboratorio académico o infraestructura interna donde exista permiso para realizar monitoreo de tráfico.

2. MODELO OSI APLICADO AL IDS

El IDS analiza información relacionada con varias capas del modelo OSI. Esto permite identificar eventos de seguridad desde diferentes niveles de comunicación en red.

┌──────────────────────────────┬──────────────────────────────────────────────────────────────┐
│ Capa OSI                     │ Aplicación dentro del IDS                                    │
├──────────────────────────────┼──────────────────────────────────────────────────────────────┤
│ Capa 2 - Enlace de datos     │ Validación de direcciones MAC mediante lista blanca.         │
├──────────────────────────────┼──────────────────────────────────────────────────────────────┤
│ Capa 3 - Red                 │ Validación de direcciones IP internas y externas.            │
├──────────────────────────────┼──────────────────────────────────────────────────────────────┤
│ Capa 4 - Transporte          │ Identificación de tráfico TCP y UDP.                         │
├──────────────────────────────┼──────────────────────────────────────────────────────────────┤
│ Capa 7 - Aplicación          │ Registro de dominios consultados mediante DNS o HTTP.        │
└──────────────────────────────┴──────────────────────────────────────────────────────────────┘

La validación de la lista blanca se relaciona principalmente con las capas 2 y 3, debido a que compara direcciones MAC e IP. El monitoreo de sitios se relaciona con la capa 7, ya que registra dominios consultados. El módulo de threat intelligence trabaja con direcciones IP externas para detectar posibles conexiones riesgosas.

5. PROTECCIÓN DE CREDENCIALES

El código fuente documentado no contiene contraseñas escritas directamente dentro del programa. Para proteger las credenciales se utiliza un archivo .env, el cual debe permanecer únicamente en el equipo donde se ejecuta el sistema.

El archivo .env almacena datos de configuración como:

SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=[correo_remitente@gmail.com](mailto:correo_remitente@gmail.com)
SMTP_PASSWORD=contraseña_de_aplicacion
ADMIN_EMAIL=[administrador@dominio.com](mailto:administrador@dominio.com)
NETWORK_INTERFACE=Wi-Fi
WHITELIST_FILE=config/whitelist.csv
BLACKLIST_FILE=config/blacklist.txt
LOG_DIR=logs
REPORT_DIR=reports
DEBUG_MODE=true

Medidas de protección aplicadas:

* El archivo .env no debe subirse al repositorio.
* En GitHub solo se debe publicar env.example como plantilla.
* Las credenciales SMTP se cargan mediante variables de entorno.
* La contraseña del correo no se escribe dentro del código fuente.
* El correo administrador se puede modificar sin cambiar el código.
* Las rutas de configuración pueden ajustarse desde .env.
* Si se usa Gmail, se recomienda usar contraseña de aplicación y no la contraseña principal de la cuenta.
* Los archivos logs y reports no deben publicarse si contienen datos de pruebas reales.

Ejemplo de uso seguro:

1. Descargar el proyecto desde GitHub.
2. Copiar env.example.
3. Renombrar la copia como .env.
4. Colocar las credenciales reales únicamente en el equipo local.
5. Ejecutar el IDS.

PSEUDOCÓDIGO DE PROTECCIÓN DE CREDENCIALES:

INICIO
cargar archivo .env
validar que existan variables obligatorias
SI falta SMTP_PASSWORD ENTONCES
mostrar error de configuración
detener ejecución
FIN SI

```
usar SMTP_USER y SMTP_PASSWORD solamente en memoria
NO imprimir contraseña en pantalla
NO guardar contraseña en logs
```

FIN

[MOD-001] ARCHIVO: ids_main.py

PROPÓSITO:

El archivo ids_main.py es el módulo principal del Sistema IDS Institucional. Su función es iniciar el programa, cargar la configuración, inicializar los módulos internos y comenzar la captura de tráfico de red.

Este archivo funciona como orquestador del sistema porque conecta los módulos de configuración, alertas, listas blancas, monitoreo, threat intelligence, forense y captura de paquetes.

IMPORTACIONES PRINCIPALES:

* os: permite identificar el sistema operativo y validar privilegios.
* sys: permite finalizar la ejecución cuando ocurre un error.
* time: puede utilizarse para control de tiempos.
* threading: permite preparar ejecución concurrente.
* logging: permite manejar niveles de registro.
* datetime: registra fecha y hora de inicio del IDS.

Módulos internos importados:

* cargar_configuracion
* ModuloListaBlanca
* ModuloMonitoreoTrafico
* ModuloThreatIntelligence
* ModuloForense
* ModuloCapturaPaquetes
* ModuloAlertas
* configurar_logger

FUNCIONAMIENTO:

1. Muestra el banner inicial del sistema.
2. Carga la configuración desde .env.
3. Inicializa el logger principal.
4. Inicializa el módulo de alertas por correo.
5. Inicializa el módulo de lista blanca.
6. Inicializa el módulo de threat intelligence.
7. Inicializa el módulo de monitoreo de tráfico.
8. Inicializa el módulo forense.
9. Inicializa el módulo de captura de paquetes.
10. Inicia la captura de tráfico.
11. Permite detener el IDS con Ctrl + C.
12. Registra errores críticos si ocurre una falla.

SUBÍNDICES EN EL CÓDIGO:

[MAIN-001] Banner de inicio
Presenta el nombre del sistema, versión, licencia y equipo de desarrollo.

[MAIN-002] Función main()
Orquesta la ejecución general del IDS.

Llama a cargar_configuracion() para obtener variables del archivo .env.

Configura el registro principal del sistema mediante configurar_logger().

[MAIN-005] Inicialización de alertas
Crea un objeto ModuloAlertas usando SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD y ADMIN_EMAIL.

[MAIN-006] Inicialización de módulos de análisis
Crea los objetos ModuloListaBlanca, ModuloThreatIntelligence, ModuloMonitoreoTrafico y ModuloForense.

[MAIN-007] Inicialización de captura
Crea el objeto ModuloCapturaPaquetes usando la interfaz configurada.

[MAIN-008] Manejo de interrupción
Detiene el IDS correctamente cuando el usuario presiona Ctrl + C.

[MAIN-009] Verificación de privilegios
Verifica si el programa se ejecuta como administrador en Windows o como root en Linux.

SEGURIDAD EN ESTE MÓDULO:

El módulo principal no contiene contraseñas en texto plano. Las credenciales se reciben desde el diccionario config, cargado previamente desde el archivo .env.

Ejemplo del fragmento relevante:

alertas = ModuloAlertas(
smtp_host     = config["SMTP_HOST"],
smtp_port     = int(config.get("SMTP_PORT", 587)),
smtp_user     = config["SMTP_USER"],
smtp_password = config["SMTP_PASSWORD"],
admin_email   = config["ADMIN_EMAIL"],

Este diseño permite cambiar las credenciales sin modificar el código fuente.

LIMITACIONES:

* Si el archivo .env no existe, el sistema no puede cargar la configuración.
* Si SMTP_PASSWORD está vacío, el módulo de correo puede fallar.
* Si NETWORK_INTERFACE está mal escrito, la captura puede no iniciar.
* Si no se ejecuta con permisos elevados, Scapy no podrá capturar paquetes.

[MOD-003] ARCHIVO: whitelist.py
───────────────────────────────────────────────────────────────────────────────

PROPÓSITO:

El módulo whitelist.py administra la lista blanca de dispositivos autorizados. Su objetivo es comparar las direcciones IP y MAC detectadas en la red contra el archivo config/whitelist.csv.

FUNCIONAMIENTO:

1. Cargar el archivo whitelist.csv.
2. Leer las IPs y MACs autorizadas.
3. Analizar los paquetes capturados.
4. Extraer IP y MAC de origen cuando sea posible.
5. Comparar esos valores contra la lista blanca.
6. Si el dispositivo no está autorizado, generar una alerta.
7. Registrar el evento en el logger.

ARCHIVO DE CONFIGURACIÓN:

config/whitelist.csv

Ejemplo de estructura:

ip,mac,descripcion
192.168.1.10,AA:BB:CC:DD:EE:FF,Equipo autorizado
192.168.1.20,11:22:33:44:55:66,Laptop administrativa

PSEUDOCÓDIGO:

INICIO ModuloListaBlanca

```
cargar archivo whitelist.csv

PARA cada línea del archivo
    obtener IP
    obtener MAC
    guardar dispositivo autorizado
FIN PARA

AL recibir paquete
    extraer IP origen
    extraer MAC origen

    SI IP o MAC no está en lista blanca ENTONCES
        registrar evento sospechoso
        enviar alerta al administrador
    SINO
        permitir tráfico como autorizado
    FIN SI
```

FIN

RELACIÓN CON MODELO OSI:

* Capa 2: validación de dirección MAC.
* Capa 3: validación de dirección IP.

SEGURIDAD EN ESTE MÓDULO:

La lista blanca ayuda a identificar dispositivos no autorizados dentro de la red. Esto permite detectar posibles conexiones ajenas a la infraestructura institucional.

FUNCIONAMIENTO:

MÓDULOS QUE RECIBEN PAQUETES:

* whitelist.py: valida IP/MAC.
* threat_intel.py: revisa IPs peligrosas.
* traffic_monitor.py: registra dominios y tráfico.
* forensics.py: apoya en análisis de IPs sospechosas.

PSEUDOCÓDIGO:

INICIO ModuloCapturaPaquetes

```

PARA cada paquete capturado
    enviar paquete a lista blanca

    detener captura

```

FIN

SEGURIDAD EN ESTE MÓDULO:

La captura de paquetes debe realizarse únicamente en redes autorizadas. Su uso fuera de una infraestructura propia o institucional puede representar un problema legal y ético.

[MOD-006] ARCHIVO: threat_intel.py

PROPÓSITO:

El módulo threat_intel.py compara direcciones IP detectadas en el tráfico contra una lista negra de IPs peligrosas. Esto permite identificar conexiones hacia destinos relacionados con malware, virus, ataques o comportamiento sospechoso.

FUNCIONAMIENTO:

2. Guardar las IPs peligrosas en memoria.
3. Analizar IPs origen y destino de los paquetes.
4. Comparar las IPs con la lista negra.
5. Si se detecta coincidencia, generar alerta.
6. Enviar información al módulo forense para ampliar el análisis.

ARCHIVO DE CONFIGURACIÓN:

config/blacklist.txt

Ejemplo:

8.8.8.8

```
cargar blacklist.txt

    guardar IP peligrosa
FIN PARA

AL recibir paquete
    extraer IP origen
    extraer IP destino

        enviar alerta al administrador
        solicitar análisis forense
    FIN SI
```

FIN

SEGURIDAD EN ESTE MÓDULO:

La lista negra permite detectar conexiones de riesgo. Sin embargo, una IP en lista negra no siempre confirma un ataque; debe interpretarse como indicador de posible riesgo que requiere revisión.

───────────────────────────────────────────────────────────────────────────────
[MOD-008] ARCHIVO: forensics.py
───────────────────────────────────────────────────────────────────────────────

PROPÓSITO:

El módulo forensics.py realiza tareas de apoyo forense cuando se detecta una IP sospechosa. Su objetivo es obtener información adicional sobre la IP para apoyar al administrador en la toma de decisiones.

FUNCIONAMIENTO:

1. Recibir una IP sospechosa.
2. Consultar información pública o servicios de reputación.
3. Obtener datos como proveedor, país, reporte de abuso o contacto abuse.
4. Generar un resumen para el administrador.
5. Enviar el resultado al módulo de alertas.

POSIBLES DATOS OBTENIDOS:

* IP analizada.
* Proveedor de servicios.
* País o región aproximada.
* Contacto abuse.
* Nivel de riesgo.
* Fecha de consulta.
* Fuente de información.

El análisis forense debe usarse como apoyo informativo. La información obtenida sobre una IP no debe utilizarse para atacar, escanear o intervenir sistemas externos. Su finalidad es únicamente defensiva y de documentación del incidente.

7. ANÁLISIS JURÍDICO DEL MONITOREO

El monitoreo de red dentro de una organización puede considerarse legal siempre que se realice bajo autorización, con fines de seguridad, dentro de infraestructura propia o administrada por la institución, y con una política clara de uso aceptable de recursos tecnológicos.

El IDS no debe utilizarse para interceptar información privada sin autorización ni para monitorear redes ajenas. Su finalidad debe limitarse a la protección de la infraestructura institucional, detección de incidentes y generación de bitácoras de seguridad.

MARCO JURÍDICO GENERAL EN MÉXICO:

1. Constitución Política de los Estados Unidos Mexicanos

El artículo 16 reconoce la protección de datos personales y establece que las personas tienen derecho a la protección de su información. Por este motivo, cualquier monitoreo que pueda involucrar datos personales debe realizarse con límites, finalidad definida y medidas de protección.

2. Ley Federal de Protección de Datos Personales en Posesión de los Particulares

Esta ley es relevante cuando las bitácoras del IDS puedan contener datos que identifiquen o hagan identificable a una persona. Por ejemplo:

* IP asignada a un usuario específico.
* MAC vinculada a un dispositivo de una persona.
* Correo electrónico del administrador.
* Registros de navegación o dominios consultados.
* Nombre del equipo o usuario asociado a una conexión.

En caso de que la organización sea particular y almacene este tipo de datos, es necesario considerar principios como finalidad, consentimiento, proporcionalidad, seguridad y confidencialidad.

3. Código Penal Federal

El Código Penal Federal contempla conductas relacionadas con acceso ilícito a sistemas y equipos de informática. Por esta razón, el IDS debe utilizarse únicamente en sistemas propios, autorizados o institucionales, evitando cualquier uso en redes externas sin permiso.

CRITERIO DE LEGALIDAD PARA EL PROYECTO:

Para que el monitoreo sea aceptable dentro de una organización, se deben cumplir las siguientes condiciones:

* La red monitoreada debe pertenecer a la institución o estar bajo su administración.
* Los usuarios deben conocer que existe monitoreo con fines de seguridad.
* El monitoreo debe limitarse a eventos técnicos necesarios.
* Las bitácoras deben protegerse contra acceso no autorizado.
* La información registrada debe usarse únicamente para seguridad informática.
* Debe existir una política interna de uso aceptable.
* Las bitácoras no deben publicarse en repositorios públicos.
* El personal responsable debe tratar la información con confidencialidad.

8. POLÍTICA DE MONITOREO Y TRATAMIENTO DE BITÁCORAS

POLÍTICA DE USO Y MONITOREO DEL SISTEMA IDS INSTITUCIONAL

1. Finalidad del monitoreo

El Sistema IDS Institucional será utilizado únicamente para fines de seguridad informática, detección de dispositivos no autorizados, identificación de conexiones hacia IPs peligrosas, generación de alertas y registro de actividad técnica relevante dentro de la red institucional.

2. Alcance

El monitoreo aplicará únicamente sobre la infraestructura de red autorizada por la organización. No se permitirá el uso del IDS en redes externas, redes públicas, redes de terceros o sistemas donde no exista autorización expresa.

3. Datos registrados

El IDS podrá registrar los siguientes datos técnicos:

* Fecha y hora del evento.
* Dirección IP de origen.
* Dirección MAC detectada.
* Dominio consultado.
* Protocolo utilizado.
* Evento detectado.
* Resultado de comparación con lista blanca o lista negra.
* Información de apoyo forense sobre IPs sospechosas.

4. Uso de la información

La información registrada será utilizada únicamente para:

* Analizar incidentes de seguridad.
* Detectar dispositivos no autorizados.
* Revisar tráfico sospechoso.
* Generar reportes internos.
* Apoyar la respuesta ante incidentes.

5. Confidencialidad

Las bitácoras generadas por el IDS deberán ser consultadas únicamente por personal autorizado. No deberán compartirse públicamente ni subirse a GitHub si contienen datos reales de usuarios, dispositivos o actividad de red.

6. Conservación de bitácoras

Las bitácoras deberán conservarse solo durante el tiempo necesario para cumplir con la finalidad de seguridad. Una vez que ya no sean requeridas, deberán eliminarse de forma segura.

7. Seguridad de credenciales

Las credenciales de correo, contraseñas SMTP y claves de API deberán almacenarse únicamente en el archivo .env local. No deberán escribirse en el código fuente ni publicarse en repositorios.

8. Aviso a usuarios

La organización deberá informar a los usuarios que la red puede ser monitoreada con fines de seguridad. Este aviso puede integrarse en una política de uso aceptable de recursos tecnológicos.

9. Prohibiciones

Queda prohibido:

* Usar el IDS en redes ajenas sin autorización.
* Capturar tráfico con fines distintos a seguridad.
* Publicar bitácoras con datos reales.
* Compartir contraseñas o claves de API.
* Modificar el IDS para realizar ataques.
* Usar la información para fines personales o no autorizados.

10. Responsable del sistema

El administrador de seguridad o responsable de TI será el encargado de configurar, ejecutar y revisar el IDS, así como proteger las bitácoras generadas.

10. NOTAS TÉCNICAS ADICIONALES

11. Uso de variables de entorno

El uso de .env permite separar la configuración sensible del código fuente. Esto facilita publicar el proyecto sin exponer credenciales.

2. Uso de lista blanca

La lista blanca permite definir qué dispositivos son autorizados dentro de la red. Si un dispositivo no aparece en la lista, el sistema puede generar una alerta.

3. Uso de lista negra

La lista negra permite detectar conexiones hacia IPs consideradas peligrosas. Esta lista puede actualizarse manualmente o con fuentes externas.

4. Bitácoras

Las bitácoras permiten revisar eventos después de la ejecución. Sin embargo, deben protegerse porque pueden contener información técnica de la red.

5. Alertas por correo

El sistema puede enviar alertas al administrador. Para esto se requiere configurar correctamente SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASSWORD y ADMIN_EMAIL.

6. Ejecución con privilegios

La captura de paquetes requiere privilegios elevados. En Windows se debe ejecutar PowerShell como Administrador. En Linux se debe utilizar sudo.

7. Uso ético

El IDS debe utilizarse únicamente con autorización. El objetivo del sistema es defensivo, académico y de apoyo al monitoreo institucional.
